Fundamentação Teórica

O que são os padrões GRASP?

GRASP — General Responsibility Assignment Software Patterns — é um conjunto de princípios que orientam como distribuir responsabilidades entre as classes de um sistema orientado a objetos.

Seu objetivo central é responder a uma pergunta simples: "Qual classe deve ser responsável por determinada operação?" A resposta a essa pergunta determina diretamente a qualidade estrutural do sistema.

Neste trabalho aplicamos cinco padrões: Information Expert, High Cohesion, Low Coupling, Pure Fabrication e Controller.

Contexto

Situação-Problema

Uma biblioteca municipal precisa de um sistema para registrar empréstimos de livros, controlar prazos de devolução e calcular multas para devoluções atrasadas. O sistema foi implementado de forma improvisada, com uma única classe concentrando todas as responsabilidades.

O sistema funciona, porém qualquer alteração em uma regra de negócio — como mudar o valor da multa por dia — exige modificações em múltiplos pontos do código, aumentando o risco de erros e dificultando a manutenção.

O Depois — Código com GRASP

Estrutura das classes

Livro

Armazena título e autor. Responsável por retornar sua própria descrição.

— Information Expert

Usuario

Armazena nome e situação de adimplência. Decide se o usuário tem penalidade extra.

— Information Expert / High Cohesion

CalculadoraMulta

Fabricação pura: não existe no domínio real, mas centraliza a regra de cálculo.

— Pure Fabrication / Low Coupling

Emprestimo

Gerencia livro, usuário e datas. Calcula apenas os dias de atraso — nada mais.

— High Cohesion / Low Coupling

O Depois — Information Expert

Classes Livro e Usuario

Livro.java expert 
class Livro {
  private String titulo;
  private String autor;

  public Livro(String titulo,
               String autor) {
    this.titulo = titulo;
    this.autor  = autor;
  }

  public String getDescricao() {
    return titulo + " — " + autor;
  }
}
Usuario.java expert 
class Usuario {
  private String  nome;
  private boolean adimplente;

  public Usuario(String nome,
                boolean adimplente) {
    this.nome       = nome;
    this.adimplente = adimplente;
  }

  public String  getNome() {return nome;}

  public boolean temPenalidadeExtra() {
    return !adimplente;
  }
}

Cada classe opera exclusivamente sobre os dados que ela mesma possui — princípio central do Information Expert.

O Depois — Pure Fabrication / High Cohesion

CalculadoraMulta e Emprestimo

CalculadoraMulta.java fabrication 
class CalculadoraMulta {
  private static final double
    MULTA_DIA   = 2.50;
  private static final double
    PENALIDADE  = 1.5;

  public double calcular(
      long diasAtraso,
      Usuario usuario) {

    if (diasAtraso <= 0) return 0;
    double multa = diasAtraso * MULTA_DIA;
    if (usuario.temPenalidadeExtra())
      multa *= PENALIDADE;
    return multa;
  }
}

O Depois — Controller

DevolucaoController

DevolucaoController.java controller 
class DevolucaoController {
    private CalculadoraMulta calculadora = new CalculadoraMulta();

    public void processarDevolucao(Emprestimo emprestimo) {
        long   diasAtraso = emprestimo.calcularDiasAtraso();
        double multa      = calculadora.calcular(diasAtraso, emprestimo.getUsuario());
        imprimirRecibo(emprestimo, diasAtraso, multa);
    }

    private void imprimirRecibo(Emprestimo emp, long dias, double multa) {
        System.out.println("======= RECIBO DE DEVOLUÇÃO =======");
        System.out.println("Usuário: " + emp.getUsuario().getNome());
        System.out.println("Livro:   " + emp.getLivro().getDescricao());
        if (multa > 0)
            System.out.printf("Multa: R$ %.2f%n", multa);
        else
            System.out.println("Devolução no prazo. Sem multa.");
        System.out.println("===================================");
    }
}

O Controller orquestra o fluxo sem concentrar regras de negócio. Ele delega o cálculo para CalculadoraMulta e a descrição para cada especialista.

Análise Comparativa

Os cinco padrões aplicados

Padrão GRASP Aplicado em Benefício
Information Expert Livro, Usuario, Emprestimo Cada classe opera sobre os dados que possui, sem depender de terceiros
High Cohesion Livro, Usuario, Emprestimo Classes com responsabilidade única, curtas e fáceis de manter
Low Coupling Emprestimo, Controller Mudança na regra de multa não afeta Emprestimo nem Usuario
Pure Fabrication CalculadoraMulta Elimina a duplicação e centraliza a regra de negócio em um único ponto
Controller DevolucaoController Separa a camada de coordenação da camada de domínio
Conclusão

O que a refatoração gerou

1
Manutenibilidade Alterar o valor da multa ou a regra de penalidade exige modificação em um único ponto do código.
2
Legibilidade Cada classe tem um nome e uma responsabilidade que se explicam por si mesmos, sem necessidade de comentários extensos.
3
Extensibilidade Adicionar um novo tipo de desconto ou um novo meio de notificação não requer reescrever as classes de domínio.
4
Testabilidade Cada classe pode ser testada de forma isolada, pois suas dependências são claramente definidas e delimitadas.
